# Phase 0: Foundation

> **Learning model:** You write the code. Claude advises, teaches, and answers questions. Scaffolding (boilerplate with no meaningful decisions) is provided on request. "You Drive" sections are where the real work and learning happen — Claude will not implement these unless explicitly asked.

**Goal:** Scaffold a working `magpie` binary with cobra CLI, XDG config loading, and 3-tier vault resolution via `.magpie/` sentinel detection.

**Architecture:** Three independent packages — `cmd` (cobra commands), `internal/config` (config structs and two-tier merge), `internal/vault` (vault resolution). Each has a single clear responsibility and can be understood and tested in isolation.

**Tech Stack:** Go 1.22+, `github.com/spf13/cobra` (CLI), `gopkg.in/yaml.v3` (config), Go standard library only beyond these two.

**Design:** [[magpie — Design Spec]] — core principles, vault structure, config format, context.md schema.

---

## Learning Goals

- Cobra's command/subcommand model and how `PersistentPreRun` propagates context
- XDG Base Directory spec — why `~/.config/magpie/` and not `~/magpie/`
- `filepath.WalkDir` for sentinel detection (walk up, not down)
- Two-struct config merge: global defaults, vault-local overrides, vault wins on conflict

## Task 3: Config Struct and Loading

**[YOU DRIVE]**

**Files:** `internal/config/config.go`, `internal/config/config_test.go`

### Interfaces

```go
// Config holds magpie configuration. All fields are optional.
type Config struct {
    DefaultVault string            `yaml:"default_vault"`
    Plugins      map[string]string `yaml:"plugins"`
    // Vault-local fields (from .magpie/config.yaml)
    InboxPath   string `yaml:"inbox_path"`
    ArchivePath string `yaml:"archive_path"`
    LogPath     string `yaml:"log_path"`
}

func Load(path string) (*Config, error)
func configPath() string
```

`Load` returns an empty Config (not an error) if the file doesn't exist — missing config is valid. `configPath` returns `$XDG_CONFIG_HOME/magpie/config.yaml` if set, otherwise `~/.config/magpie/config.yaml`.

### What to test

- `Load` with a valid YAML file returns correct fields
- `Load` with a missing file returns an empty Config, no error
- `Load` with malformed YAML returns an error
- `configPath` returns `~/.config/magpie/config.yaml` when `XDG_CONFIG_HOME` is unset
- `configPath` returns `$XDG_CONFIG_HOME/magpie/config.yaml` when set

## Task 6: Vault Resolution

**[YOU DRIVE]**

**Files:** `internal/vault/resolve.go`, `internal/vault/resolve_test.go`

### Interface

```go
// ErrNoVault is returned when no vault can be found via any resolution method.
var ErrNoVault = errors.New("no vault found: set MAGPIE_VAULT, add default_vault to config, or run from inside a vault")

// Resolve finds the vault path using the 3-tier hierarchy:
//  1. MAGPIE_VAULT environment variable
//  2. Walk upward from cwd, find first directory containing .magpie/
//  3. default_vault in global config (~/.config/magpie/config.yaml)
func Resolve(cwd string) (string, error)
```

> [!question] Decisions to make while implementing
> - How far does the CWD walk go? (stopping at `$HOME` avoids surprising matches)
> - What does "contains `.magpie/`" mean — must it be a directory, not a file?
> - If `MAGPIE_VAULT` is set but the path doesn't exist, is that an error or do you trust it?

### What to test

```go
func TestResolve(t *testing.T) {
    // Hint: use t.TempDir() to create temp vault structures
    // Hint: set/unset MAGPIE_VAULT with t.Setenv("MAGPIE_VAULT", path)
}
```

- `MAGPIE_VAULT` set → returns that path (even without `.magpie/`)
- CWD is inside a vault → finds vault root by walking up
- CWD is nested several directories deep inside a vault → still finds root
- CWD is outside any vault, no config → returns `ErrNoVault`
- CWD is outside any vault, `default_vault` set in config → returns that path
- Two nested vaults (`.magpie/` at both levels) → returns the innermost one

## What's Next

Phase 1 (Vault I/O) covers `inbox`, `archive`, and `log` commands — and wires **plugin dispatch** alongside them. `Resolve` and `Merge` from Phase 0 will be called by every command from here on.

Plugin dispatch (`syscall.Exec` for unknown subcommands) ships in Phase 1, not Phase 4. Phase 4 adds the management commands (`plugin install/remove/status`). The dispatch mechanism itself is a small addition to `cmd/root.go` — do it early so every subsequent phase can be tested against real plugin binaries.

> [!tip]
> Before starting Phase 1, ask Claude to review your `Resolve` implementation — the walk logic and error handling are where subtle bugs tend to hide.
